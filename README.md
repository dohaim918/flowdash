## 0. 프로젝트 개요 (간단 요약)

본 프로젝트는 효율적인 일정 관리를 위한 사용자 맞춤형 투두 리스트(Todo List) 기반의 칸반형 태스크 관리 대시보드를 구현하는 과제입니다.

<br>

칸반 보드 방식의 태스크 CRUD 기능을 중심으로 기간별 필터링, 실시간 통계 제공, 개인화된 테마 설정 등을 포함하였으며, 

팀 협업을 통해 확장성을 고려한 데이터 구조와 객체 지향적 로직을 설계·구현하였습니다.

## 2. 수행 절차 및 방법 (Process & Strategy)

### 2-1. 진행 순서

1. **요구사항 전체 리뷰 및 필수 기능 정의**
* 요구사항 전체 검토 및 핵심 구현 기능 상세 정의.
* 팀원과 논의하여 UI/CSS 및 JS 로직 파트별 담당 영역 분담.

2. **데이터 모델 및 LocalStorage Key 설정**
* Task 객체 구조(ID, 제목, 내용, 우선순위, 생성일 등) 및 데이터 형식 정의.
* Todo 데이터 구조와 LocalStorage Key 요구사항에 맞춰 설정.

3. **보드 구조 및 렌더링 방식과 디자인 결정**
* 보드 렌더링 방식 및 글래스모피즘 기반 디자인 요소 확정.
* 모달, 버튼, 입력창 등 공통 UI 컴포넌트 디자인 가이드 수립.

4. **UX / 테마 구현 및 컨테이너 쿼리 적용**
* 사용자 편의성을 고려한 테마 색상 및 배경 효과 구현.
* 컨테이너 쿼리 위주로 활용하여 컴포넌트 단위의 반응형 레이아웃 적용.

5. **필터 / 정렬 / 검색 파이프라인 구현**
* 기간 필터, 우선순위 필터, 정렬, 검색어 필터를 개별 함수로 설계.
* 함수를 통해 모든 필터링 로직이 순차적으로 작동하는 파이프라인 구축.
* 데이터 변경 시 필터 함수를 거쳐 화면이 재렌더링되는 구조 완성.

6. **통합 작업 및 최종 QA**
* 팀원별 작업 결과물 병합 및 기능 단위 테스트 수행.
* 코드 교차 검증을 통해 예외 케이스 처리 및 버그 수정.

### 2-2. 협업 규칙
- 브랜치 전략: main / dev / feature/*
- PR 단위: 기능 1개 기준
- PR 전 rebase 필수
- 스펙 변경은 dev merge 전까지만 허용

## 3. 프로젝트 구조 및 아키텍처

### 3-1. 디렉터리 구조
```
flowdash/
├─ README.md
├─ index.html
├─ css/
│  ├─ reset.css              * 기본 스타일 초기화 
│  ├─ style.css              * 전반적인 레이아웃 및 메인 스타일 
│  ├─ delete.css             * 삭제 관련 UI 스타일 
│  ├─ modal_style.css        * 모달 공통 스타일 
│  └─ theme.css              * 테마 변수 및 다크 모드 
│
├─ img/
│
└─ js/
   ├─ main.js                * DOM 요소 탐색, 로컬스토리지 CRUD, 로드 로직, 주요 함수 관리
   ├─ deleteTodo.js          * 투두 삭제 로직 및 삭제 모달 제어
   ├─ filter.js              * 검색 및 필터링 파이프라인
   ├─ todoCard.js            * 투두 카드 생성 및 렌더링
   ├─ top.js                 * 카드 상단(제목, 내용, 날짜, 테마) 관련 로직 
   └─ ui/                      (icons.js : svg / modal.js : 모달기능 / todoCard.js :)
      ├─ card.js               (top.js : 제목,내용,날짜)
      ├─ modal.js            * 모달 공통 동작 로직
      └─ icons.js            * SVG 아이콘을 변수로 관리 

       ※ todoCard.js : 카드생성 Ui + Dom + 모달역할 + 통계 
```

### 3-2. 모듈 책임 분리
- `main.js` : 이벤트 리스너 관리
- `modal.js` : Ui 생성과 모달 + 통계 관리
- `deleteTodo.js` : 앱의 상태인 Todo 리스트를 로컬 스토리지와 동기화하면서 관리
- `icons.js` : svg 관리
- `filter.js` : 검색 파이프라인 전담 관리 
- `todoCard.js` : Todo 앱에서 카드 생성, 모달, UI 상태, 통계 계산까지 관리
- `top.js` : 전역 UI 상태 관리 

### 3-3. 데이터 흐름  

main.js (요소 탐색 및 로컬스토리지 저장 요청)
↓
ui/ 컴포넌트 (카드 생성 및 보드 배치)
↓
최종 화면 (리스트 렌더링 + 대시보드 갱신)
↓
filter.js (파이프라인으로 데이터 가공)

## 6. 트러블슈팅 (Troubleshooting) > 팀원별 문제점 추가 <

### 6-1. Dropdown
- 증상: 드롭다운 커스텀 X / 정렬이 안됨 
- 원인: option으로 하면 커스텀이랑 정렬이 어렵다.
- 해결: ul il 활용하여 커스텀과 정렬 한번에 해결
- 회고: 앞으로 UI/UX 관련 기능은 초기 설계 단계에서 사용자 경험과 커스터마이징 가능성을 더 꼼꼼하게 고려해야한다는 점 

### 6-2. Filter
- 증상: 검색파이브라인 버튼 클릭 시 다른 이벤트 날아감
- 원인: 걸어둔 곳에 조건이 빠져있고 몇개는 주석처리
- 해결: 조건을 한곳에만 넣어두고 서로 상호작용 하는 부분에 안 넣음
- 회고: 주석처리라던가 조건이나 코드를 한두 번 더 체크하는 습관..

### 6-3. 조건
- 증상: 버튼 filter 화면에 보이는 글자로 했더니 에러
- 원인: 화면에 보이는 글자로 하면 첫번째 요소 텍스트만 가져옴
- 해결: 객체 데이터로 해서 값을 만들어서 불러와서 쓰기
- 회고: 아직 응용이라던가 지식이 많이 부족한 점이 많아 더욱 공부..

### 6-4. 필터링 
- 증상 : 필터링 파이프라인이나 UI 컴포넌트 생성 중 `ReferenceError` 또는 `TypeError`가 빈번하게 발생하여 기능이 멈춤.
- 원인 : 반복문 내 변수 참조 실수나 노드리스트(NodeList)와 개별 요소를 혼동하여 잘못된 메서드를 호출하는 등 복잡한 데이터 흐름 속에서 로직이 꼬임.
- 해결 : 브라우저 개발자 도구의 Console 탭을 활용해 단계별로 데이터를 출력하며 변수 정의 유무와 데이터 타입을 실시간으로 확인함. 에러 메시지가 가리키는 지점을 추적하여 잘못된 참조를 바로잡음.
- 회고 : 단순히 코드를 작성하는 것보다 에러 메시지를 읽고 콘솔로 데이터 변화를 추적하는 과정이 더 중요하다는 것을 체득함. 이를 통해 로직이 복잡해질수록 직관에 의존하기보다 콘솔을 통한 단계별 검증이 필수임을 배움.
